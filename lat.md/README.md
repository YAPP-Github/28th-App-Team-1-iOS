# lat.md — 도메인 지식 볼트 & lat 방법론

> 제품이 **이렇게 설계된 의도**와 **코드**를 같은 흐름에 두고, **검증 여부**를 늘 추적 가능하게 두는 곳.
> PM·디자이너·개발이 같은 도메인 언어로 보는 단일 지식 볼트(Obsidian).

이 폴더는 "코드가 무엇을 하는가"가 아니라 **"이 코드가 어떤 의도·도메인과 연결되는가"** 를 적는다.
역할이 다른 문서는 자리도 다르다:

| 무엇 | 어디 |
|---|---|
| 도메인 지식·의도·숨은 의존 | **`lat.md/`** (여기) |
| 심볼/API 레퍼런스·튜토리얼 | DocC `Projects/App/Documentation/Architecture.docc` |
| 첫 빌드·환경·세팅 | `docs/` |
| 기획서 → 아키텍처 매핑 **작업 문서** | `docs/work/` |
| 작업 세부 규칙 | `CLAUDE.md` · 팀 컨벤션 `CONTRIBUTING.md` |

---

## 목표

**의도 ↔ 코드 ↔ 검증** 세 가지를 끊기지 않게 연결한다.

```
기획/디자인 의도  ──@lat──▶  코드  ──@lat-test(예정)──▶  검증
       ▲                                                 │
       └──────────── lat.md 설명서 + domain.map ──────────┘
```

## 왜 도입하나

> "코드 곳곳에 색인 꼬리표를 붙이고, 그 꼬리표를 설명서·관계지도와 연결해서,
> 나중에 누가(특히 AI가) 봐도 **'이거 고치면 어디가 영향받는지'** 즉시 알 수 있게 만든 정리법."

코드는 시간이 지나면 거대한 회사처럼 복잡해진다. "이 업무가 누구 소속이지?", "이걸 바꾸면 어느 팀이 영향받지?" 를 아무도 한눈에 모르게 된다.

게다가 우리는 **AI와 함께 단기간에** 제품을 만든다. AI는 빠르게 많은 코드를 내놓지만 같은 의도에도 매번 조금씩 다른 코드를 내고, 사람이 100% 검수하기는 어렵다. 그래서 **"의도한 것"과 "실제 코드" 사이의 틈**이 모르는 사이 쌓인다. 이건 팀 크기 문제가 아니라 **코드 생성 속도가 사람의 이해 속도를 앞지르기** 때문이다.

lat.md 는 코드마다 "어떤 의도·도메인과 연결되는지" 꼬리표를 붙여 그 연결이 끊기지 않게 한다. 여기에 "사람이 검증했는지" 상태까지 적으면, **아직 검증 안 된 AI 코드만 한 줄 검색으로 골라낼 수 있다.**

## 어떻게 동작하나 — 4가지 장치

### 1. `lat.md/` 폴더 = 도메인 설명서 모음 (백과사전)

코드는 빽빽해서 사람이 읽기 어렵다. 그래서 별도 설명서 폴더를 둔다. 여기서 **"도메인"은 업무 영역**(회사로 치면 부서)이라고 이해하면 된다.

- `architecture.md` — 시스템 전체 설계와 "왜 이렇게 했는가" (총론)
- `users.md` / `profile.md` 등 — 각 업무 영역(도메인)의 설명서

### 2. `@lat` 주석 = 코드에 붙인 색인 라벨

코드 파일 안에 다는 메모(주석)에 특별한 꼬리표를 단다. **이 레포는 이미 이 규칙으로 라벨링되어 있다.**

```swift
// Projects/Feature/Users/Sources/UsersFeature.swift
// @lat: [[users#Profile Edit Handoff]]
// depends-on: [[profile#Save]], [[app#Cross-feature Routing]]
```

- `@lat:` — "이 코드는 어느 도메인의 어느 부분 소속인가"
- `depends-on:` — "무엇에 의존하는가(없으면 안 돌아가 / 바꾸면 영향받는다)"
- `[[ ]]` — 위키피디아 하이퍼링크처럼 "누르면 그 설명서로 연결된다"

> 검증 연결용 `@lat-test: [[tests#…]]` 라벨은 **아직 미도입**이다. 테스트 추적이 필요해지면 같은 규칙으로 추가한다.
>
> 라벨 종류·배치·유지 규칙의 **상세 스펙은 [labeling.md](labeling.md)** — Claude 가 코드에 주석을 달 때 따르는 기준.

도서관 책마다 "역사 코너 3번 선반, 관련 책 ○○" 색인 라벨을 붙여둔 것과 같다.

### 3. ripgrep 검색 = 색인을 이용한 초고속 찾기

라벨이 있으니 키워드 한 줄로 관련 코드를 즉시 찾는다. Makefile 단축 명령:

```bash
make lat q=profile        # profile 과 엮인 코드 전부 ([[profile 검색 → delegate 의존도 잡힘)
make lat-deps q=profile   # profile 을 바꾸면 영향받는 곳 (depends-on)
make lat-all              # 코드 전체의 @lat 라벨 목록
```

(ripgrep 있으면 `rg`, 없으면 `grep` 자동 폴백. 속도 위해 `brew install ripgrep` 권장.)

### 4. `domain.map.md` = 부서 간 협업 관계도(조직도)

꼬리표가 "코드 한 줄 단위의 작은 연결"이라면, **"도메인끼리 큰 틀에서 어떻게 협업하는가"** 같은 큰 그림은 [`domain.map.md`](domain.map.md) 에 따로 그린다. 작은 연결(꼬리표)과 큰 연결(관계 지도)을 둘 다 갖춰 부분과 전체를 모두 본다.

이 레포의 핵심 예시: **Users ↔ Profile** 은 서로 `import` 하지 않는데도 협업한다 (delegate + AppFeature 중재). import 그래프엔 안 보이는 이 의존이 바로 관계도가 필요한 이유다.

## 도입 준비

### 전원 공통

- **공통 도메인 용어집** — 같은 용어를 같은 의미로. ("주문"이 PM에겐 장바구니, 개발자에겐 결제 직전이면 혼란.) 문서·코드·라벨의 기준이며 **최우선 준비물.**
- **협업 리듬 합의** — 학습/설계/논의 주기(주간 권장)와 각 단계 미팅 주체를 명확히.
- **툴 계정·권한** — Claude 공용 계정, Jira/Confluence/Figma/Notion 등 시작일에 전원 접근 확보.
- **정보 단일 위치 약속** — 학습 자료는 Confluence, 제품·코드 지식은 Claude/lat.md, 화면 설계는 Figma 처럼 위치를 하나로.

### PM

- **도메인 정의** — 제품을 업무 영역(주문/결제/회원…)으로 나누고 경계를 명확히. 이게 `lat.md/` 폴더 구조와 1:1 로 연결된다. 비개발 도메인 정의가 흐리면 지식 그래프도 흐려진다.
- **마크다운 PRD** — 워드/PPT 대신 마크다운으로 작성해야 Claude·코드와 연동된다. "배경 → 목표 → 사용자 → 흐름 → 의사결정·트레이드오프" 고정 틀.
- **트레이드오프 기록** — "A 대신 B를 택한 이유와 포기한 △△" 를 문서화. 논의의 핵심 자료가 된다.

### 디자이너

- **사용자 학습 계획** — 인터뷰 등 대상·방법·빈도를 명확히. "매주 학습"이 선언에 그치지 않게 첫 주 대상·방법을 킥오프 전에 확정.
- **Figma 화면·설명 규칙** — 화면별 설명 형식을 사전에 합의해 개발자가 혼동 없이 코드로 옮기게.
- **학습 정리 루틴** — 매주 학습을 같은 양식으로 Confluence 에 정리해 PM·개발이 참고하게.

### Dev

- **lat.md 폴더 구조 설계** — PM 도메인 정의에 맞춰(비개발 정의 선행 필수). 전원이 함께.
- **`@lat` 주석 컨벤션 합의** — 라벨 형식 표준화로 검색 실패 방지.
- **코드 관리 규칙** — 브랜치/네이밍/PR/리뷰/머지/릴리즈.
- **검색 환경 통일** — ripgrep 등 도구 환경을 모두 동일하게.
- **`domain.map.md` 초안** — PM 도메인 정의 기반으로 개발 관점에서 관계를 구체화.
- **PRD ↔ 코드 연결 규칙** — 마크다운 PRD의 특정 부분이 어떤 코드·주석과 연결되는지.

## 주의점 — 성패는 두 연결에 달려 있다

1. **PM의 도메인 정의 → 개발의 lat.md 구조**
2. **디자이너의 사용자 학습 → PM·개발의 설계 입력**

비개발이 "대충" 넘기면 개발이 추측으로 메우고, 그 순간부터 이해도가 갈라진다. 그래서 비개발의 준비는 **"내 산출물이 곧 개발의 입력"** 이라는 의식이 핵심이다. 반대로 개발은 비개발 문서의 공백/모순을 확인하고 **적극적으로 되물어야** 한다. 그리고 `@lat` 주석을 **빠짐없이, 같은 규칙으로** 연결한다.
